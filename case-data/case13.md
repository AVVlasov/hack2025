## **1. Концепция проекта**

### Описание кейса / проблемы

```
Текущая ситуация:

ООО «Инженерно-производственный центр» ежедневно получает 
от клиентов заявки на поставку газопроводного оборудования. 
Заявки приходят в различных форматах: Word, Excel, PDF, 
отсканированные документы.

Проблемы текущего процесса:

1. РАЗНОРОДНОСТЬ ФОРМАТОВ
   - Каждый клиент использует свой шаблон заявки
   - Технические параметры указаны в разных форматах:
     * Текстовые таблицы
     * Отсканированные чертежи
     * Встроенные изображения с характеристиками
     * Смешанные документы (текст + картинки)

2. РУЧНАЯ ОБРАБОТКА ЗАНИМАЕТ МНОГО ВРЕМЕНИ
   - Менеджер тратит 30-45 минут на одну заявку
   - Необходимо вручную:
     * Извлечь технические характеристики
     * Распознать параметры из изображений
     * Сопоставить с номенклатурой производителя
     * Найти соответствие в ТУ (Технических Условиях)
     * Заполнить шаблон для 1С ЕРП
     * Создать "Заказ клиента" в системе

3. ВЫСОКИЙ ПРОЦЕНТ ОШИБОК
   - Человеческий фактор: опечатки, неверное сопоставление
   - Ошибки приводят к:
     * Поставке неправильной продукции
     * Возвратам и рекламациям
     * Потере времени на переделку
     * Снижению лояльности клиентов

4. МАСШТАБИРОВАНИЕ ЗАТРУДНЕНО
   - При росте заказов требуется увеличивать штат
   - Обучение нового сотрудника занимает 2-3 недели
   - Зависимость от экспертизы конкретных людей

Примеры реальных проблем:
- Заявка на 50 позиций обрабатывается 3-4 часа
- В месяц 15-20% заявок требуют переделки из-за ошибок
- Клиенты ждут КП 1-2 дня вместо нескольких часов
```

### Описание образа результата

```
Интеллектуальная система автоматической обработки 
технических заявок:

ЭТАП 1: ПРИЁМ ЗАЯВКИ
→ Клиент отправляет заявку на email или загружает 
  через веб-форму
→ Система автоматически определяет формат документа

→ OCR-движок распознаёт текст и изображения

  - Наименование изделия
  - Диаметр, давление, материал, длина и т.д.
  - Количество
  - Дополнительные требования

ЭТАП 3: СОПОСТАВЛЕНИЕ С НОМЕНКЛАТУРОЙ
→ Система анализирует извлечённые параметры
→ Сопоставляет с базой ТУ производителя

→ Уведомление о неоднозначных позициях

Ключевые метрики результата:

## **3. Функциональность**

### MVP (Минимально жизнеспособный продукт)

```
МОДУЛЬ 1: ПРИЁМ И РАСПОЗНАВАНИЕ ДОКУМЕНТОВ

1.1. Загрузка документов
   - Web-интерфейс для загрузки файлов (drag-n-drop)
   - Приём заявок через email (интеграция с почтой)
   - Поддержка форматов: PDF, DOCX, XLSX, JPG, PNG
   - Автоматическое определение типа документа

1.2. OCR и распознавание
   - Извлечение текста из PDF и Word
   - OCR для отсканированных документов и изображений
   - Распознавание таблиц и структурированных данных
   - Извлечение изображений из документов для анализа

1.3. Извлечение технических параметров (NLP)
   - Определение наименования изделия
   - Извлечение ключевых параметров:
     * Диаметр (DN)
     * Давление (PN)
     * Материал (сталь, чугун, полиэтилен и т.д.)
     * Длина, тип соединения
     * ГОСТ / ТУ
   - Распознавание количества
   - Извлечение дополнительных требований

---

МОДУЛЬ 2: СОПОСТАВЛЕНИЕ С БАЗОЙ ТУ ПРОИЗВОДИТЕЛЯ

2.1. База знаний номенклатуры
   - Загрузка и хранение ТУ производителя
   - Структурированная база изделий с параметрами
   - Поиск по характеристикам
   - Версионирование базы ТУ

2.2. Интеллектуальное сопоставление
   - Точное совпадение параметров
   - Поиск аналогов при неточном совпадении
   - Учёт допустимых отклонений параметров
   - Scoring (оценка уверенности) для каждого сопоставления

2.3. Обработка исключений
   - Выявление несовпадений и конфликтов
   - Флаг "требует проверки специалиста"
   - Предложение альтернатив с обоснованием
   - Уведомление менеджера о неоднозначных позициях

---

МОДУЛЬ 3: ФОРМИРОВАНИЕ ЗАКАЗА

3.1. Генерация шаблона Excel
   - Заполнение шаблона производителя
   - Автоматическое форматирование
   - Экспорт в формат, совместимый с 1С ЕРП
   - Добавление артикулов и кодов продукции

3.2. Интеграция с 1С ЕРП
   - API-подключение к 1С (REST/SOAP)
   - Автоматическое создание документа "Заказ клиента"
   - Заполнение карточки клиента
   - Создание позиций заказа с ценами
   - Статус обработки: "Черновик" для проверки

3.3. Предпросмотр и контроль качества
   - Визуальный интерфейс для проверки результата
   - Подсветка позиций с низкой уверенностью (scoring < 80%)
   - Возможность ручной корректировки
   - Сравнение исходной заявки и результата
   - Утверждение и отправка в 1С

---

МОДУЛЬ 4: АДМИНИСТРАТИВНАЯ ПАНЕЛЬ

4.1. Dashboard для менеджеров
   - Список заявок: новые, в обработке, проверка, завершённые
   - Статистика: обработано за день/неделю/месяц
   - Время обработки каждой заявки
   - Процент автоматизации (без участия человека)

4.2. Управление базой ТУ
   - Загрузка новых ТУ (Excel, XML)

   - Добавление синонимов и альтернативных названий
   - Поиск и фильтрация номенклатуры

4.3. Настройки и конфигурация

   - Настройка почтового сервера для приёма заявок
   - Шаблоны Excel для разных типов продукции
   - Правила валидации и пороги уверенности (scoring)

4.4. История и логирование
   - История обработки каждой заявки
   - Логи изменений и корректировок
   - Аудит действий пользователей

---

ТЕХНИЧЕСКИЕ КОМПОНЕНТЫ MVP:

- Backend: Python (FastAPI/Django)
- OCR: Tesseract, Google Cloud Vision API, или ABBYY
- NLP: spaCy, Named Entity Recognition (NER)
- База данных: PostgreSQL
- Очереди: Redis / RabbitMQ

### Use-cases (сценарии использования)

```
USE-CASE 1: Стандартная заявка

1. Клиент отправляет Excel-файл на email zakaz@ipc.ru
2. Система автоматически получает письмо и загружает вложение
3. Извлекает данные из таблицы
4. Сопоставляет с базой ТУ (все позиции найдены, scoring > 90%)
5. Формирует заказ в 1С ЕРП автоматически
6. Отправляет уведомление менеджеру: "Заказ готов для 
   утверждения"
7. Менеджер проверяет (2 минуты) и утверждает
→ Итого: 5 минут вместо 40

---

USE-CASE 2: Сложная заявка с неоднозначностями

1. Клиент загружает PDF через веб-интерфейс
2. Система распознаёт 15 позиций
3. Для 12 позиций найдено точное совпадение (scoring > 90%)
4. Для 3 позиций scoring = 65-75% (требуется проверка)
5. Система помечает эти позиции красным флажком
6. Менеджер получает уведомление
7. Проверяет 3 спорные позиции (5 минут), корректирует
8. Утверждает заказ для создания в 1С
→ Итого: 12 минут вместо 45

---

USE-CASE 3: Заявка с изображениями

1. Клиент присылает Word-документ с таблицей и вставленными 
   фото изделий
2. Система извлекает текст и изображения
3. OCR распознаёт текст на изображениях (параметры на 
   шильдиках)
4. NLP-модуль извлекает параметры из обоих источников
5. Сопоставление с ТУ
6. Менеджер проверяет и утверждает
→ Итого: 10 минут вместо 60
```
